Code Ahead

cordova.exec();


cordova.exec(win, fail, service,
             action, args);
  • win: success callback
  • fail: failure callback
  • service: native class that will do the work
  • action: native method that will do the work
  • args: arguments to pass into native method

How Does PhoneGap Map Service+Action to Native Code?

config.xml document

Mapping Service+Action


<plugin name="SERVICE!" value="native class identifier"/>

e.g. (Android)


<plugin name="Accelerometer"
        value="org.apache.cordova.AccelListener"/>

PhoneGap Init Sequence

  1. Each <plugin> is loaded during PhoneGap's bootup

  2. Once all plugins are loaded, main PhoneGap event deviceready is fired
  3. Calls to cordova.exec are routed to appropriate class

Down the Rabbit Hole We Go:


Native Code

Step 1: Native Interface

  • Each platform has CordovaPlugin class / interface
  • Extend it to make your own PhoneGap plugin!

Step 2: Native Methods

  • JS' cordova.exec invokes Plugin method
  • PhoneGap keeps track of invocation via a Callback ID
  • Expects method to return a Result: error, data (JSON), nothing, whatever.
  • Method can Keep The Callback (more on this later)

Android

PhoneGap Plugin interface: CordovaPlugin.java <-- extend this.

  • Has hooks into lifecycle events
  • All cordova.exec calls route through 
    public boolean execute(String action, JSONArray args, CallbackContext callbackContext)

Android Contacts Search Example


public boolean execute(String action, JSONArray args,
                       final CallbackContext callbackContext) {
    if (action.equals("search")) {
        final JSONArray filter = args.getJSONArray(0);
        final JSONObject options = args.getJSONObject(1);
        this.cordova.getThreadPool().execute(new Runnable() {
            public void run() {
                JSONArray res = contactAccessor.search(filter,
                                                       options);
                callbackContext.success(res);
            }
        });
    }
}
